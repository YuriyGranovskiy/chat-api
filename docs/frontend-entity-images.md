# Изображения сущностей (World, Persona, Location) — заметки для фронтенда

Документ описывает изменения бэкенда: одна картинка на мир, персону или локацию, хранение на сервере, новые поля в JSON и эндпоинты.

## Что изменилось в данных

В ответах списков и там, где сериализуются сущности через модели `WorldData`, `PersonaData`, `LocationData`, добавлено опциональное поле:

| Поле        | Тип            | Описание |
|------------|----------------|----------|
| `image_url` | `string \| null` | Относительный URL картинки. Если `null` — изображения нет. |

Пример фрагмента объекта мира:

```json
{
  "id": "01HZ...",
  "name": "Мой мир",
  "description": "...",
  "image_url": "/api/media/abc123...token..."
}
```

Чтобы показать картинку в браузере, к `image_url` нужно добавить базовый URL API (origin бэкенда), например: `https://api.example.com/api/media/...` или тот же хост, с которого ходите к `/api`.

**Важно:** `GET` по `image_url` **не требует** JWT. Доступ обеспечивается длинным случайным токеном в пути. Обычный `<img src="...">` без заголовка `Authorization` работает.

---

## Загрузка и замена изображения

Один и тот же метод используется и для первой загрузки, и для замены файла.

| Метод | Путь | Auth |
|-------|------|------|
| `PUT` | `/api/worlds/{worldId}/image` | Bearer JWT |
| `PUT` | `/api/personas/{personaId}/image` | Bearer JWT |
| `PUT` | `/api/locations/{locationId}/image` | Bearer JWT |

**Тело запроса:** `multipart/form-data`, поле файла: **`file`**.

Поддерживаемые типы (по содержимому и/или `Content-Type`): PNG, JPEG, WebP, GIF. Ограничение размера по умолчанию: **10 МБ** (настраивается на бэкенде).

**Успех:** HTTP `200`, пустое тело.

**Ошибки (примеры):**

- `400` — нет файла, пустой файл, слишком большой файл, неподдерживаемый тип. Тело JSON: `{ "error": "текст причины" }`.
- `404` — сущность с таким id не найдена. Тело: `{ "error": "World not found" }` (или Persona / Location).

После успешной загрузки имеет смысл заново запросить список или сущность, чтобы получить актуальный `image_url` (токен в пути при **первой** загрузке создаётся и дальше не меняется при замене файла — URL для `<img>` остаётся стабильным).

---

## Удаление изображения

| Метод | Путь | Auth |
|-------|------|------|
| `DELETE` | `/api/worlds/{worldId}/image` | Bearer JWT |
| `DELETE` | `/api/personas/{personaId}/image` | Bearer JWT |
| `DELETE` | `/api/locations/{locationId}/image` | Bearer JWT |

**Успех:** HTTP `200`, пустое тело. В БД обнуляются поля картинки; старый `image_url` перестаёт работать (будет 404).

**Ошибки:** `404`, если сущность не найдена.

---

## Получение файла изображения (прямая ссылка)

| Метод | Путь | Auth |
|-------|------|------|
| `GET` | `/api/media/{token}` | не нужен |

`{token}` — это последний сегмент из `image_url` (значение после `/api/media/`).

Ответ: бинарное тело изображения с корректным `Content-Type`. При отсутствии файла или неизвестном токене: `404` и JSON `{ "error": "Not found" }`.

---

## Пример: загрузка из браузера

```ts
async function uploadWorldImage(worldId: string, file: File, accessToken: string) {
  const form = new FormData();
  form.append("file", file);

  const res = await fetch(`${API_BASE}/api/worlds/${worldId}/image`, {
    method: "PUT",
    headers: {
      Authorization: `Bearer ${accessToken}`,
    },
    body: form,
  });

  if (!res.ok) {
    const err = await res.json().catch(() => ({}));
    throw new Error(err.error ?? res.statusText);
  }
}
```

Для персон и локаций замените путь на `/api/personas/{id}/image` и `/api/locations/{id}/image`.

---

## Поведение при удалении сущности

При удалении мира, персоны или локации через существующие `DELETE /api/worlds/{id}` (и аналоги) файл изображения на диске удаляется вместе с записью. Отдельно вызывать `DELETE .../image` не обязательно.

---

## Резюме для интеграции

1. Отображать превью: брать `image_url` из списка миров/персон/локаций, префиксовать базовым URL API, вставлять в `img` / `background-image`.
2. Загрузка: `PUT` + `multipart/form-data`, поле `file`, заголовок JWT.
3. Сброс картинки: `DELETE` на `.../image` с JWT.
4. Не кэшируйте навсегда без стратегии: при замене файла по тому же URL отдаётся новое содержимое; при полном удалении картинки URL становится недействительным.

Если понадобится уточнить базовый URL или CORS в вашем окружении — это настройки деплоя бэкенда; сам API префикс для этих маршрутов — `/api`.
